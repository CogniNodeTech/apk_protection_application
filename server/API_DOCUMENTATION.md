# SafeGuard Threat Intelligence Backend API Documentation

Welcome to the SafeGuard Threat Intelligence API documentation. This service serves as the secure backend for the SafeGuard Android security application. It manages threat detection lookups, client database syncs, user accounts, and telemetry processing.

## 🌐 Global Configuration & Headers

*   **Production URL**: `https://api4.cogninodeapk.in`
*   **Default Dev URL**: `http://127.0.0.1:3000`
*   **Android Emulator Dev Loopback**: `http://10.0.2.2:3000`

### Required Headers

| Header | Type | Required | Description |
| :--- | :--- | :--- | :--- |
| `Authorization` | String | Yes (for `/v1/*`) | Bearer token authorization. Format: `Bearer <token>` |
| `X-Request-Id` | String | Optional | Unique trace identifier. Generated by Nginx if omitted. Used for log aggregation. |
| `Content-Type` | String | Yes (for POST) | Must be `application/json` |

## 🛡️ API Rate Limiting & Protections

To protect the backend from abuse, automated scraping, and DDoS attempts, the server enforces distinct rate-limiting rules:

*   **Verification Rate Limit**: Configured via `RATE_LIMIT_MAX_REQUESTS` (default: `30` requests per `60` seconds).
*   **Threat Feed Rate Limit**: Configured via `THREAT_FEED_RATE_LIMIT_MAX_REQUESTS` (default: `12` requests per `3600` seconds).
*   **Replay Request Detection**: Verification requests with identical package, hash, and timestamp parameters within a 10-minute window are blocked with a `409 Conflict` response to prevent MITM replays.

### 2.2 Bulk Threat-Feed Sync
*   **Route**: `GET /v1/threat-feed`
*   **Access**: Bearer Auth Required
*   **Query Parameters**:
    *   `since` (integer): Optional Unix-epoch millisecond timestamp cursor. Only returns threat signatures discovered *after* this timestamp.
    *   `limit` (integer): Optional limit of signatures returned (Max `1000`, default `200`).
*   **Description**: Returns bulk list of signatures to update the client device's local offline database.
*   **Response Payload (`200 OK`)** — *Unsigned Mode*:
    ```json
    {
      "items": [
        {
          "sha256": "4a0815181938efe6a17b077a56162391277a1670ac615020f5c3e792bf1627fa",

          "threat_name": "Spyware.SpyNote",
          "threat_family": "SpyNote",
          "severity": 95,
          "first_seen_ms": 1773167400000,
          "source": "malwarebazaar"
        }
      ],
      "next_cursor_ms": 1773167400000,
    ```

## 5. Error Handling & Status Codes

The server API uses standard HTTP status codes:

| Status Code | Name | Description |
| :--- | :--- | :--- |
| `200` | OK | The request completed successfully. |
| `400` | Bad Request | Missing parameter, invalid formatting, or stale timestamps. |
| `401` | Unauthorized | Missing or malformed `Authorization: Bearer` header. |
| `403` | Forbidden | Invalid API Token or expired authentication token. |
| `409` | Conflict | Triggered when duplicate verify request / replay attack is detected, or when signing up an email that already exists. |
| `422` | Unprocessable Entity | Payload validation failed (e.g. invalid string length or extra fields provided). |
| `429` | Rate Limit Exceeded | Request limit exceeded. Standard responses include a `Retry-After` header. |
| `502` | Bad Gateway | Threat provider (MalwareBazaar) is unavailable. |
