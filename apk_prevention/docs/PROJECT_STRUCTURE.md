## SafeGuardApp Project Structure

This document describes the **folders, modules, and key files** in the SafeGuard APK Protection application.  
It is meant as an orientation guide for new contributors and a reference for ongoing development.

---

## 1. Root Project: `SafeGuardApp`

- **File** `settings.gradle.kts`  
  - Configures Gradle plugin and dependency repositories (`google`, `mavenCentral`, `jitpack`).  
  - Sets `rootProject.name = "SafeGuardApp"`.  
  - Declares Gradle modules:
    - `:app` – main Android app module (UI & entry point).
    - `:core` – domain logic and orchestration.
    - `:data` – persistence and networking.
    - `:security` – protection layers and integrity checks.
    - `:mlmodel` – ML model integration.
    - `:notification` – notifications abstraction.
    - `:education` – user tutorials/education flows.

- **File** `build.gradle.kts`  
  - Top-level Gradle configuration.  
  - Declares core plugins (Android, Kotlin, Hilt, KSP, Detekt) with `apply false`, so subprojects can opt-in.  
  - Configures **Detekt** to automatically apply to Kotlin Android subprojects.  
  - Defines a `clean` task to delete the root `build` directory.

- **Folders** (high level)
  - `app/` – Android app/UI & Hilt wiring.
  - `core/` – business/domain logic and orchestration.
  - `data/` – database, preferences, repositories, remote API.
  - `security/` – layered security engine and quarantine.
  - `mlmodel/` – TensorFlow Lite integration and feature extraction.
  - `notification/` – notification helper module.
  - `education/` – tutorial/education logic.
  - `docs/` – project documentation and implementation plans.
  - `.gradle/` – Gradle internal cache and metadata (generated, not hand-edited).
  - `*/build/` – per-module build outputs (generated by Gradle).

## 3. `app/` – Main Android Application Module

`app` contains the Android entry point, UI (screens and components), app-specific DI, and background workers.

### 3.1. `app/src/main/`

- **File** `AndroidManifest.xml`  
  - Declares `SafeGuardApplication` and `MainActivity`.  
  - Requests necessary permissions (e.g. storage, package inspection).  
  - Registers services, workers, and any exported components.

#### 3.1.1. Application & Navigation

- **File** `kotlin/com/safeguard/SafeGuardApplication.kt`  
  - Custom `Application` class, likely annotated with Hilt (e.g. `@HiltAndroidApp`).  
  - Initializes global dependencies, telemetry, and crash reporting.

- **File** `kotlin/com/safeguard/MainActivity.kt`   
  - Main activity hosting the app’s navigation and top-level UI.  
  - Ties the Compose theme, navigation graph, permission handling, and overall app chrome together.

- **File** `kotlin/com/safeguard/ui/navigation/AppNavigation.kt`  
  - Central navigation graph: defines routes for Dashboard, History, Quarantine, Protection Status, Scan Results, Detailed Analysis, and Settings.  
  - Wires each route to its corresponding composable screen and handles deep links/arguments.

#### 3.1.2. UI Screens

- **Folder** `kotlin/com/safeguard/ui/screens/dashboard/`
  - **File** `DashboardScreen.kt` – Composable for the main dashboard, showing current protection state, quick actions, and recent scan summaries.  
  - **File** `DashboardViewModel.kt` – Aggregates data (risk, latest scans, upcoming scans) from core/domain layers and exposes it to the dashboard UI.

- **Folder** `kotlin/com/safeguard/ui/screens/history/`
  - **File** `HistoryScreen.kt` – Screen for viewing historical scan results and security events.  
  - **File** `HistoryViewModel.kt` – Loads and filters scan history from the `ScanRepository` (via the data layer).

- **Folder** `kotlin/com/safeguard/ui/screens/quarantine/`
  - **File** `QuarantineScreen.kt` – UI showing quarantined APKs, with options to restore or delete.  
  - **File** `QuarantineViewModel.kt` – Manages quarantined items, talking to the `QuarantineRepository` and `QuarantineManager`.

- **Folder** `kotlin/com/safeguard/ui/screens/protectionstatus/`
  - **File** `ProtectionStatusScreen.kt` – High-level view of protection state across layers (file monitoring, hash checks, permissions, signatures, ML, cloud).  
  - **File** `ProtectionStatusViewModel.kt` – Provides aggregate protection state from core orchestration and layer results.  
  - **File** `ProtectionStatusDetailScreen.kt` – Detailed per-layer status and metrics.  
  - **File** `ProtectionStatusDetailViewModel.kt` – Holds detailed data for each protection layer and handles “drill-down” interactions.

- **Folder** `kotlin/com/safeguard/ui/screens/scanresults/`
  - **File** `ScanResultsScreen.kt` – Screen showing the result of a specific scan (verdict, recommendations, threat details).  
  - **File** `ScanResultsViewModel.kt` – Owns state for a single scan result, including recommended actions and user choices.

- **Folder** `kotlin/com/safeguard/ui/screens/detailedanalysis/`
  - **File** `DetailedAnalysisScreen.kt` – Deep-dive view showing per-layer findings, permissions, signatures, ML output, and cloud checks for one APK.  
  - **File** `DetailedAnalysisViewModel.kt` – Gathers and exposes the detailed data for the selected APK to the UI.

- **Folder** `kotlin/com/safeguard/ui/screens/settings/`
  - **File** `SettingsScreen.kt` – Screen for adjusting app settings: scan schedule, notifications, trusted apps, telemetry, theme, etc.  
  - **File** `SettingsViewModel.kt` – Reads and writes settings via `SecurePreferencesManager` and repositories.

#### 3.1.3. UI Components & Theme

- **Folder** `kotlin/com/safeguard/ui/components/`
  - **File** `RiskGauge.kt` – Composable gauge component (e.g. meter or dial) visualizing risk scores.  
  - **File** `HexagonThreatLevel.kt` – Hexagonal visual indicator for threat level, potentially representing multiple layers.  
  - **File** `BottomNavBar.kt` – Bottom navigation bar UI to switch between top-level screens.

  - **File** `ProtectionStatusCard.kt` – Summary card for protection status, reused on dashboard and status screens.

  - **File** `SecureActionButton.kt` – Styled button emphasizing secure or high-impact actions.

- **Folder** `kotlin/com/safeguard/ui/theme/`

  - **File** `Type.kt` – Typography definitions (font families, weights, styles).

#### 3.1.4. Services, Workers & App-Level Domain

- **Folder** `kotlin/com/safeguard/service/`
  - **File** `FileObserverService.kt` – Service monitoring APK-related file changes and triggering scans.

## 4. `core/` – Domain & Orchestration Module

`core` contains business/domain logic, models, repository interfaces, telemetry/crash interfaces, and orchestration components.

### 4.1. Domain Models & Interfaces

- **Folder** `core/src/main/kotlin/com/safeguard/core/domain/model/`
  - **File** `APKFile.kt` – Domain representation of an APK (path, package info, metadata).  
  - **File** `ScanResult.kt` – Aggregate scan result with overall verdict and per-layer information.  
  - **File** `ThreatInfo.kt` – Represents detailed info about a threat (type, severity, description).  
  - **File** `UserDecision.kt` – Encodes user choices (trust, quarantine, delete, ignore).  
  - **File** `Action.kt` – Represents actions the system will perform in response to a verdict.  
  - **File** `Verdict.kt` – Canonical verdict enumeration or sealed type (e.g. SAFE, SUSPICIOUS, MALICIOUS, UNKNOWN).  
  - **File** `LayerResult.kt` – Domain representation of a single protection layer’s result (status, findings, timing).

- **Folder** `core/src/main/kotlin/com/safeguard/core/domain/repository/`
  - **File** `ScanRepository.kt` – Interface for storing and retrieving scan history.  
  - **File** `ThreatDatabaseRepository.kt` – Interface abstracting access to local threat data/signatures.  
  - **File** `CloudVerificationRepository.kt` – Interface for cloud-based supplement checks and reputation.  
  - **File** `QuarantineRepository.kt` – Interface for persisting and retrieving quarantine records.

- **Folder** `core/src/main/kotlin/com/safeguard/core/domain/usecase/`
  - **File** `ScanAPKUseCase.kt` – Contract for executing an end-to-end scan of an APK.  
  - **File** `QuarantineAPKUseCase.kt` – Contract for quarantining APKs according to domain rules.

- **Folder** `core/src/main/kotlin/com/safeguard/core/domain/telemetry/`
  - **File** `ScanTelemetry.kt` – Interface for emitting telemetry events/metrics from scan flows.

- **Folder** `core/src/main/kotlin/com/safeguard/core/domain/crash/`
  - **File** `CrashReporter.kt` – Interface defining how crashes are captured and reported.

- **Folder** `core/src/main/kotlin/com/safeguard/core/domain/layer/`
  - **File** `ProtectionLayer.kt` – Abstraction for individual protection layers (input contract and expected `LayerResult`).

### 4.2. Orchestration

- **Folder** `core/src/main/kotlin/com/safeguard/core/orchestration/`
  - **File** `ScanOrchestrator.kt` – Orchestrates running multiple `ProtectionLayer`s, aggregating their outputs into a unified `ScanResult`.  
  - **File** `RiskAssessmentEngine.kt` – Computes risk scores/classifications based on layer results, history, and threat data.  
  - **File** `ZeroTrustDecisionEngine.kt` – Applies zero-trust decision rules using context and risk signals to decide allow/deny/quarantine.

## 6. `security/` – Security Engine Module

`security` defines the protection layers, quarantine logic, and runtime integrity checks.

### 6.1. Module Manifest

- **File** `security/src/main/android/AndroidManifest.xml`  
  - Module-specific manifest entries (if any), merged into the app manifest.

### 6.2. Integrity & Quarantine

- **Folder** `security/src/main/kotlin/com/safeguard/security/integrity/`
  - **File** `RuntimeEnvironmentChecker.kt` – Inspects Android runtime environment for compromise indicators (root, debug builds, unsafe settings).  
  - **File** `AntiDebugChecker.kt` – Detects debugger attachment or instrumentation-based tampering.

- **Folder** `security/src/main/kotlin/com/safeguard/security/quarantine/`
  - **File** `QuarantineManager.kt` – Manages physical handling of quarantined APKs (moving, storing, restoring, deleting).

### 6.3. Protection Layers

- **Folder** `security/src/main/kotlin/com/safeguard/security/layers/base/`
  - **File** `ProtectionLayer.kt` – Base implementation utilities for concrete protection layers (shared execution, error handling).

- **Folder** `security/src/main/kotlin/com/safeguard/security/layers/layer1/`
  - **File** `FileSystemMonitor.kt` – Monitors filesystem for APK-related changes.  
  - **File** `Layer1FileMonitor.kt` – Layer-1-specific file monitoring logic.  
  - **File** `Layer1Result.kt` – Result type for layer 1 checks.

- **Folder** `security/src/main/kotlin/com/safeguard/security/layers/layer2/`
  - **File** `HashValidator.kt` – Computes SHA-256 + SHA-512 (single I/O pass) and a TLSH fuzzy hash, then compares against the local malware/trusted DB. Surfaces collision evidence: a SHA-256 hit whose stored SHA-512 mismatches the local one is downgraded to **SUSPICIOUS @ 0.85** with `isCollision=true`; a confirming SHA-512 adds explicit "SHA-512 confirmed" evidence on the MALICIOUS verdict.  
  - **File** `Layer2Result.kt` – Result of hash validation. Carries `sha256`, `sha512`, fuzzy-hash similarity, and the `isCollision` flag for downstream decision logic and forensics.

- **Folder** `security/src/main/kotlin/com/safeguard/security/layers/layer3/`
  - **File** `PermissionAnalyzer.kt` – Analyzes manifest permissions for over-privileged or suspicious requests.  
  - **File** `Layer3Result.kt` – Result of permission analysis.

- **Folder** `security/src/main/kotlin/com/safeguard/security/layers/layer4/`
  - **File** `SignatureValidator.kt` – Validates developer signatures using whitelists/blacklists (e.g. `signature_blacklist.txt`).  
  - **File** `Layer4Result.kt` – Result of signature validation.

- **Folder** `security/src/main/kotlin/com/safeguard/security/layers/layer5/`
  - **File** `MLAnalyzer.kt` – Performs ML-based threat detection by calling into the `mlmodel` module.  
  - **File** `Layer5Result.kt` – Result of ML analysis (scores, classification).

- **Folder** `security/src/main/kotlin/com/safeguard/security/layers/layer6/`
  - **File** `CloudVerifier.kt` – Contact point for cloud verification and reputation checks, typically via the `data` module.  
  - **File** `Layer6Result.kt` – Result of cloud verification.

- **Folder** `security/src/main/kotlin/com/safeguard/security/layers/layer7/`
  - **File** `Layer7Result.kt` – Result of YARA-style content scanning (matched rule names, severity, evidence).
  - **File** `YaraScanner.kt` – Runs the bundled YARA-subset ruleset against an APK's `classes*.dex`, `AndroidManifest.xml`, and small native libraries (with size and per-entry caps to avoid DoS from malformed ZIPs).
  - **File** `YaraRuleSet.kt` – Loads bundled `*.yar` rules from `security/src/main/assets/yara/` once per process and exposes them to the scanner.
  - **Folder** `yara/`
    - **File** `YaraRule.kt` – Domain model for a parsed YARA rule (metadata, strings, condition AST).
    - **File** `YaraRuleParser.kt` – Pure-Kotlin tokenizer + recursive-descent parser for the supported YARA subset (literal `ascii`/`wide`/`nocase` strings, hex byte sequences with `??` wildcards, `any/all/N of them`, `and`/`or`/`not`); refuses anything outside the subset rather than silently shadowing it.
    - **File** `YaraMatcher.kt` – Evaluates parsed rules against in-memory byte buffers without allocations on the hot path.

- **Folder** `security/src/main/assets/yara/`
  - **File** `android_bankers.yar` – Bundled rules for accessibility-overlay bankers (Anatsa, Hydra, GodFather, generic banker overlay heuristics).
  - **File** `android_droppers_evasion.yar` – Bundled rules for droppers, BadPack ZIP-tamper, Frida, and AVS evasion patterns.

## 10. Generated and Build Artifacts

These folders are auto-generated by Gradle and should generally **not** be edited manually:

- `app/build/`, `core/build/`, `data/build/`, `security/build/`, `mlmodel/build/`, `notification/build/`, `education/build/`  
  - Contain compiled classes, processed resources, generated sources (e.g. Hilt, KAPT), APK outputs, and intermediate artifacts.

- `.gradle/`  
  - Gradle’s local cache and metadata for builds.

These directories are typically excluded from version control and can be safely regenerated by cleaning and rebuilding the project.
